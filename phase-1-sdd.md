# Phase 1: SDD — 先搞清楚"要做什么"

> **在没有写清楚规格之前，不要让 AI 写任何代码。规格是你跟 AI 之间的"合同"。**

## 怎么做

1. 用一两句话描述你要做什么（概述）
2. 列出功能需求，每条一句话，编号 FR1、FR2、FR3……
3. 列出验收标准——怎么判断"做完了"？编号 AC1、AC2、AC3……
4. 想清楚边界情况：没数据怎么办？输入错误怎么办？极端值怎么办？

## 产出物：规格文档

创建一份规格文档，至少包含以下内容：

```markdown
# spec.md — {项目名称}

## 概述
{一句话描述项目目标和价值}

## 功能需求
- FR1: {功能描述}
- FR2: {功能描述}
- FR3: {功能描述}

## 验收标准
- AC1: {验收条件——明确、可验证}
- AC2: {验收条件}

## 边界情况
- 当 {情况} 时，应该 {行为}
- 当 {情况} 时，应该 {行为}

## 约束条件（可选）
- 技术约束：{语言/框架限制}
- 业务约束：{必须遵守的规则}
```

> 需要更详细的模板？参考 [templates/sdd-spec-template.md](templates/sdd-spec-template.md)

## ✅ 完成检查

- [ ] 每个功能需求都能在验收标准中找到对应的验证方式？
- [ ] 边界情况至少覆盖了「正常流程」「空数据/无数据」「错误输入」三种？
- [ ] 规格里有没有出现具体技术实现（框架名、库名等）？—— 有的话暂放，留给后续阶段
- [ ] 每条需求都足够具体，AI 不会产生歧义？

全部 ✅ 后，进入 [Phase 2](phase-2-ddd.md)。（简单功能可跳过 Phase 2，直接进入 [Phase 4](phase-4-tdd.md)）
