# Phase 5: AI 实现 — 把前面的产出"喂"给 AI

> **这是最关键的一步。前面四个 Phase 的所有产出，都是为这一步准备的输入。输入质量决定输出质量。**

## 提示词构建模板

按以下结构组织你的提示词，把 Phase 1-4 的产出依次填入：

```
## 角色
你是一个 {语言/框架} 开发者。请严格按照以下规格实现代码。

## 规格文档（来自 Phase 1）
{粘贴 spec.md 的完整内容}

## 领域模型（来自 Phase 2，如有）
{粘贴统一语言定义和领域模型}

## 行为场景（来自 Phase 3，如有）
{粘贴 BDD 场景——如果跳过了 Phase 3，可省略此节}

## 测试用例（来自 Phase 4）
{粘贴 Phase 4 生成的测试代码——确认测试全部失败 🔴 后再开始实现}

## 约束条件
- 只实现规格中列出的功能，不要添加额外功能
- 不要过度设计，保持代码简单直接
- 所有测试必须通过
- {如果有 DDD 模型：严格遵循上述领域模型和分层架构}
```

## 执行策略：逐任务推进（最重要！）

**不要一次让 AI 生成所有代码。** 按测试用例逐个推进：

```
对每个测试任务：
1. 把当前任务的测试代码发给 AI
2. 让 AI 只实现让「这个测试」通过的最少代码
3. 运行这个测试，确认通过（🟢 Green）
4. 运行全部已有测试，确认没有破坏之前的功能
5. 当前任务通过后，进入下一个任务
```

## AI 写错了怎么办

| 问题 | 对策 |
|------|------|
| 测试不通过 | 把测试失败信息完整发给 AI，让它修复。如果 3 次还修不好，把这个任务拆成更小的子任务 |
| AI 添加了规格之外的功能 | 明确告诉 AI："请删除 xxx 功能，它不在规格文档中" |
| 生成的代码过于复杂 | 告诉 AI："这段代码过度设计了，请用最简单的方式重写，只满足测试要求" |
| AI 修改了不该改的代码 | 告诉 AI："请不要修改 xxx 文件/函数，只修改 yyy 部分" |

## 全部完成后

1. 运行完整测试套件 —— 必须全部通过
2. 逐条对照验收标准（Phase 1 的 AC1、AC2…）检查 —— 必须全部满足
3. 检查是否有规格之外的多余功能 —— 有则删除
4. 检查代码结构是否清晰、不过度设计 —— 不清晰则要求 AI 重构
